# Git Branching Strategies

Git branching strategies define **how teams create, manage, and merge branches** in a repository.
They help coordinate development, maintain stable releases, and enable collaboration without conflicts.

### Workflow

* Scheduled releases
* Strong testing and QA processes

## ðŸŒ± 2. GitHub Flow (Simple & Continuous)

A lightweight workflow focused on continuous deployment.

Used widely in modern web development and agile teams.

### Structure

* One long-lived branch â†’ **main**
* Short-lived feature branches for all work

### Workflow

1. Create a branch from main.
2. Make changes and commit.
3. Open a pull request for review and testing.
4. Merge into main.
5. Deploy immediately.

### Best suited for

* Fast iteration
* Continuous deployment
* Small to medium teams
* Web applications

Originally popularized by **GitHub**.

## ðŸŒ³ 3. GitLab Flow (Environment-Based)

Combines structure with continuous delivery using environment branches.

### Typical branches

* main
* staging
* production

Code moves forward through deployment environments rather than release branches.

### Best suited for

* DevOps pipelines
* Teams with staging environments
* Continuous delivery with controlled releases

Popularized by **GitLab**.

## ðŸ”€ 4. Trunk-Based Development (High Integration)

All developers work off a single branch (the trunk), usually main.

### Key principles

* Very short-lived feature branches
* Frequent merges
* Strong automated testing
* Feature flags for incomplete work

### Best suited for

* High-performing engineering teams
* CI/CD-driven development
* Microservices architecture

---

## ðŸŒ¿ 5. Release Branching (Version Maintenance)

Used when maintaining multiple versions of software simultaneously.

### Example branches

* main
* release/1.0
* release/2.0

Each version can receive independent bug fixes and support.

### Best suited for

* Enterprise software
* Libraries and frameworks
* Long-term support (LTS) products
